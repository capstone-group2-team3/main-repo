## 3. Conflict Plan

### 3.1 Process for resolving disagreement

When a technical or interpersonal disagreement appears, the team will first name the exact point of disagreement clearly. Each teammate involved will explain their view briefly and respectfully. The team will look for a consensus decision based on project goals, deadline, technical risk, and user value. If consensus is not reachable, the team will choose a compromise that everyone can live with and document the decision in Slack or GitHub.

---

### 3.2 If one teammate is dominating or steamrolling

**First step:**  
We will address it respectfully during the daily sync or in a private message. The goal is not to blame the teammate, but to restore balance and make sure every voice is heard.

**Escalation if the first step does not work:**  
If the behavior continues, the team will hold a short live discussion and agree on clearer decision rules, task ownership, and speaking time. If the problem still continues, we will escalate to the Leadership Team through Slack.

---

### 3.3 If one teammate is under-contributing or missing check-ins

**First step:**  
We will contact the teammate privately and respectfully to ask if they are blocked, confused, unavailable, or need support. We will offer help, pair programming, or a smaller task that still contributes real value.

**Escalation if the first step does not work:**  
If the teammate continues missing check-ins or not contributing, the team will document the issue and escalate to the Leadership Team through Slack. The goal is to protect the team’s progress while still giving the teammate a fair chance to re-engage.

---

### 3.4 Handling mismatched skill levels

We expect different teammates to be stronger in different technical areas. More-experienced teammates will coach by explaining decisions, reviewing code, and pairing when needed, but they will not take over the entire task. Less-experienced teammates will ask for help early instead of disappearing into a rabbit hole. A teammate who is blocked for more than 30–60 minutes should post in Slack and ask for help.

---

### 3.5 Escalation path

If our team cannot resolve a conflict internally, we will send a Slack message to the Leadership Team.

**Team agreement:**  
Every teammate agrees to this escalation path.

## 5. Work Plan

### 5.1 Task identification, assignment, tracking

We will break the project into tasks during the first working block by translating the project proposal features into a clear task list. Each task will be written as a GitHub Project card with a clear owner, expected output, and acceptance criteria.

We will track work in **GitHub Projects** only, so the team has one shared source of truth. The project board will use these columns: **Backlog**, **In progress**, **In review**, and **Done**.

A task is considered done only when:
1. The related code or document is pushed to a feature branch.
2. A Pull Request is opened.
3. The PR receives the required teammate reviews.
4. The PR is merged to main.
5. The teammate gives a brief update or demo at the next sync.

---

### 5.2 Anti-siloing note

Every teammate must be able to explain every part of the project by Demo Day. No teammate should finish an important piece alone in a black box.

Our safeguards:
1. Each teammate will pair or shadow at least once during the sprint on a task outside their primary strength area.
2. Every PR needs teammate review before merging to main.
3. Each teammate will explain their completed work during daily syncs.
4. We will document important decisions in the README, GitHub issues, PRs, or Slack.

**Team agreement:**  
Our team agrees to the anti-siloing approach.

---

### 5.3 No-solo-committing rule

Working alone on the project during non-working hours or over the weekend without a clear task assignment is not acceptable. It creates risk because the rest of the team may not understand the work and may not be able to explain it during Demo Day.

If a teammate wants to work solo for learning, they can use a scratch branch that does not merge to main unless the team reviews it and agrees. All graded project work must go through the agreed task assignment, PR review, and merge process.

**Team agreement:**  
Our team agrees to the no-solo-committing rule.

### 6.2 Branch strategy

**main:** Production-ready branch. All PRs land here after review , Two approval review.  
**dev:** Optional staging branch. We will use it only if integration becomes risky; otherwise, we prefer feature branch to main through PR review.  
**Feature branches:** Use `feature/<slug>` or `<teammate-github>/<slug>`. One task per branch. Delete branches after merge.

---

### 6.3 PR review workflow

Every PR to `main` requires **2 teammate approvals** before merge. This will be enforced using branch protection on `main`.

Rules:
1. The PR author does not merge their own PR until two other teammates approve.
2. Two other teammates must review and approve the PR.
3. Stale approval dismissal is enabled. If new commits are pushed after approval, the approval resets and teammates re-review the updated code.
4. PR title format:

- `feature/eval-harness: add held-out set loader`
- `feature/rag-api: connect retriever to FastAPI endpoint`
- `feature/docker-compose: add backend and vector database services`

PR body must include:
- What changed
- Any teammate action needed
- Test evidence, such as a screenshot or pasted output from `pytest -v`

No teammate should bypass branch protection. If an emergency requires a temporary rule change, the team must open a Slack thread and wait for Lead Instructor approval.

---

### 6.4 Merge cadence

We will merge to `main` at least once per day during the sprint when reviewed work is ready. We will also merge on-demand when a PR unblocks another teammate. When a PR is ready, the author will post in the Slack channel, tag reviewers, and mention what needs special attention.

---

- In review

Every task is a card. Every card has an owner before it moves to In progress. No unassigned cards should stay in In progress.
